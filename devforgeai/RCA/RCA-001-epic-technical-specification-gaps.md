# RCA-001: Epic Document Technical Specification Gaps

**Date:** 2026-01-27
**Severity:** HIGH
**Reporter:** Bryan

**Status:** IDENTIFIED

## Issue Description

### What Happened

Epic documents (EPIC-001 and EPIC-002) generated by devforgeai-ideation skill lacked sufficient technical specifications for cross-session story creation. A fresh Claude session running `/create-story` would not have access to:

1. **CLI command syntax** - Exact flags (`-i`, `-r`, `--context N|full`, `--format`, `--signatures`)
2. **JSON output schemas** - Structure of API responses or file formats
3. **SQLite schema** - Table definitions (Symbol, File, Metadata)
4. **File path mappings** - Which features map to which source-tree.md locations
5. **Context file references** - No frontmatter links to constitutional constraint files
6. **Requirements spec reference** - No link to detailed requirements document
7. **Crate naming errors** - Epic said "tree-sitter-languages" but dependencies.md has individual crates (tree-sitter-python, tree-sitter-typescript, etc.)
8. **Platform-specific behavior** - Unix socket vs Windows named pipe not specified

### Expected Behavior

Epic documents should be **self-contained specifications** that enable a fresh Claude session to create detailed stories with precise acceptance criteria, without needing conversational context from the original ideation session.

### Actual Behavior

Epic documents were high-level business abstractions requiring the creating session's contextual knowledge to fill in technical gaps.

### Impact

- **Severity:** HIGH
- **Affected Workflow:** Cross-session story creation
- **Consequence:** Stories created from epics would have ambiguous acceptance criteria, requiring extensive back-and-forth questions to clarify technical details
- **Workaround Applied:** Manually added Technical Specification sections to both epics during this session

---

## 5 Whys Analysis

### Why #1: Why did the epic documents lack technical specifications?

**Answer:** The ideation skill's artifact-generation.md (Phase 6.1-6.3) does not instruct to add a "Technical Specification" section to epic documents.

**Evidence:**
- **File:** `.claude/skills/devforgeai-ideation/references/artifact-generation.md` (689 lines)
- **Search:** `Grep(pattern="Technical Specification", path="artifact-generation.md")` returned no matches
- **Finding:** The workflow loads epic-template.md and populates 12 constitutional sections, but does not add technical implementation details
- **Excerpt (lines 15-46):** Section Compliance Checklist lists 12 required sections - "Technical Specification" is NOT one of them

### Why #2: Why doesn't artifact-generation.md include technical specification instructions?

**Answer:** The constitutional epic-template.md (loaded in Phase 6.1) only defines high-level sections. It doesn't require detailed technical specifications like CLI syntax, JSON schemas, or SQLite schemas.

**Evidence:**
- **File:** `.claude/skills/devforgeai-orchestration/assets/templates/epic-template.md` (265 lines)
- **Lines 109-131:** "Technical Considerations" section shows placeholders: `[New technologies/libraries to evaluate or adopt]`, `[Changes to existing architecture]`
- **Finding:** Template encourages listing technologies at a high level but not specifying exact CLI commands, schemas, or implementation details
- **Significance:** Template is intentionally abstract (epic-level planning) without story-level technical precision

### Why #3: Why doesn't the template require story-level technical details?

**Answer:** Epic documents were designed for business-level planning, assuming technical details would be added during story creation by devforgeai-story-creation skill in a SINGLE continuous session (not cross-session).

**Evidence:**
- **File:** `.claude/skills/devforgeai-ideation/references/artifact-generation.md`
- **Lines 49-59:** "Cross-Session Context Requirements" section
- **Excerpt:**
  ```
  When another Claude session resumes epic generation, it needs:
  - Brainstorm document
  - Complexity score
  - Epic decomposition
  - Feasibility assessment
  - Project context files
  ```
- **Finding:** Lists requirements for RESUMING epic generation, NOT for CONSUMING epic for story creation
- **Gap:** No mention of "epic must contain technical specifications for downstream story creation"
- **Significance:** Framework assumes linear workflow: ideation → architecture → story creation in one session

### Why #4: Why was cross-session story creation not considered?

**Answer:** The framework's original design assumed linear workflow execution in single sessions, not branching workflows where users run `/ideate` in one session, then `/create-story` in a completely separate session days later.

**Evidence:**
- **File:** `.claude/skills/devforgeai-ideation/references/completion-handoff.md` (721 lines)
- **Pattern:** Recommends `/create-context` as next action, assumes user continues immediately
- **Additional Evidence:** artifact-generation.md line 327 checks `context_files` existence to determine mode, but doesn't instruct to ADD `context_files` list to epic frontmatter for future session discovery
- **Finding:** No documentation addresses "epic as standalone artifact for independent consumption"
- **Architectural assumption:** Conversational context carries forward → epics are workflow checkpoints, not standalone specifications

### Why #5 (ROOT CAUSE): Why did the framework design assume single-session workflows?

**ROOT CAUSE:** The DevForgeAI framework was originally designed with **implicit context flow** (conversational continuity) rather than **explicit artifact-based context** (document-driven resumability). Epic documents were treated as workflow artifacts for the current session rather than standalone specifications for future sessions.

**Evidence:**
- **Architectural pattern:** Commands pass context via conversation markers (`**Business Idea:**`, `**Project Mode:**`) not via document fields
- **Template design:** epic-template.md has no `context_files:` or `requirements_ref:` fields in frontmatter (contrast with story-template which has more comprehensive metadata)
- **Workflow assumption:** `/ideate` → `/create-context` → `/create-story` happens sequentially in one session
- **Missing pattern:** No "Cross-Session Handoff Requirements" checklist specifying what epics need for independent consumption
- **Documentation gap:** No example of Session A runs `/ideate`, Session B runs `/create-story` workflow

**Validation:**
- ✅ **Would fixing prevent recurrence?** Yes - adding technical specifications and frontmatter links makes epics self-contained
- ✅ **Explains all symptoms?** Yes - all gaps (CLI syntax, schemas, file paths, context references) stem from epics not being designed for standalone consumption
- ✅ **Within framework control?** Yes - we can update template, artifact-generation.md, and validation
- ✅ **Evidence-based?** Yes - grep searches and file reads prove specifications are not required by current workflow

## Technical Specification

> **CRITICAL FOR STORY CREATION:** This section contains exact specifications required for acceptance criteria. Stories MUST reference these specs.

### CLI Interface Specification

```bash
# Document exact command syntax with all flags
{command_name} {subcommand} <ARGS> [OPTIONS]

Options:
  --flag <VALUE>    Description
  -f, --flag        Short and long forms

{command_name} example1

### API/Output Schema Specification

```json
{
  "field": "value",

    "schema": "definition"
  }

```

### Data Schema Specification

Document database schemas, file formats, or data structures:

```sql
CREATE TABLE entity_name (
    id INTEGER PRIMARY KEY,
    field TEXT NOT NULL
);
```

### File Path Mappings (per source-tree.md)

| Component | File Path | Responsibility |
|-----------|-----------|----------------|
| {Component 1} | {path from source-tree.md} | {what it does} |
| {Component 2} | {path from source-tree.md} | {what it does} |

### Behavior Specifications

Document non-obvious behavior, edge cases, or cross-cutting concerns:

- **{Behavior 1}:** {Exact specification}
- **{Behavior 2}:** {Exact specification}

### Dependencies (per dependencies.md)

| Feature | Package/Crate | Version | Purpose |

---
```

```markdown
### Step 6.1.5: Populate Technical Specification Section

**Purpose:** Fill epic's Technical Specification section with implementation details for cross-session story creation.

**This step ensures epics are self-contained documents** that future Claude sessions can consume without conversational context.

**Extraction Logic:**

```
FOR each epic being generated:

  # 1. CLI Interface Specification (if CLI tool)
  IF project_type == "CLI" OR phase_2_captured_cli_details:
    Extract from Phase 2:
      - Command syntax (base command + subcommands)
      - All flags with short/long forms
      - Arguments (required vs optional)
      - Examples with expected behavior

    Format as:
      ```bash
      {command} {subcommand} <ARGS> [OPTIONS]

      Options:
        -f, --flag <VALUE>    Description

      Examples:
        {command} example1
        {command} example2 --flag value
      ```

  # 2. Output/API Schema Specification
  IF phase_2_captured_output_format:
    Extract from Phase 2:
      - JSON structure (query, results, metadata)
      - Field types and descriptions
      - Example with realistic data

    Format as JSON schema or example

  # 3. Data Schema Specification
  IF phase_2_captured_data_entities:
    Extract from Phase 2:
      - Table definitions or document structure
      - Primary keys, foreign keys, indexes
      - Field types and constraints

    Format as SQL DDL or NoSQL structure

  # 4. File Path Mappings
  Read(file_path="devforgeai/specs/context/source-tree.md")

  FOR each feature in epic.features:
    Map feature components to source-tree.md locations:
      - Module/package directory
      - Main implementation file
      - Test file location
      - Related files

    Create table:
      | Component | File Path | Responsibility |

  # 5. Behavior Specifications
  Extract from Phase 2 NFRs and requirements:
    - Edge case handling (unsupported files, errors, empty results)
    - Platform-specific behavior (Unix vs Windows)
    - Performance behavior (progress bars, chunking)
    - Error handling patterns

  Format as bullet list with exact specifications

  # 6. Dependencies
  Read(file_path="devforgeai/specs/context/dependencies.md")

  FOR each feature in epic.features:
    List required packages/crates:
      - Package name (exact as in dependencies.md)
      - Version (from dependencies.md)
      - Purpose (why needed for this feature)

  Create table:
    | Feature | Package | Version | Purpose |

  # 7. Insert into Epic
  technical_spec_section = format_technical_spec(
    cli_syntax, schemas, file_paths, behaviors, dependencies
  )

  Insert technical_spec_section into epic after "Out of Scope" section
```

**When This Step Executes:**
- **Timing:** After epic template loaded and 12 constitutional sections populated, before epic written to disk
- **Frequency:** Once per epic (during epic generation loop)
- **Conditional:** Can skip subsections if data not applicable (e.g., no CLI for backend API project)

**Error Handling:**
```
IF technical details unclear (e.g., CLI syntax not defined in Phase 2):
  Add placeholder:
    "### CLI Interface Specification

    **To be determined during story creation.** Requires:
    - Command structure decision
    - Flag naming conventions
    - Example usage patterns"

  Better to document gaps explicitly than omit silently
```

**Validation:**
After population:
- [ ] Technical Specification section exists
- [ ] At least 3 subsections populated (not empty)
- [ ] File paths reference source-tree.md locations
- [ ] Dependencies match dependencies.md exactly
- [ ] Schemas match Phase 2 requirements elicitation
```

**Rationale:**
- Phase 2 requirements elicitation already asks about CLI commands, output formats, data models
- This information exists in session memory - just needs to be extracted and formatted into epic
- artifact-generation.md is where epic population happens - logical place for this extraction step
- Builds on existing "Cross-Session Context Requirements" concept (lines 49-59), extending it to story creation scenario

**Testing:**
1. Add Step 6.1.5 to artifact-generation.md
2. Run full `/ideate` workflow on test CLI project

   - CLI syntax from Phase 2 questions
   - JSON schema from output format question

   - Dependencies from dependencies.md
4. Close Claude session
5. Open fresh session

7. Verify story AC can reference CLI flags, JSON schema, file paths from epic without asking

Requirements (explicit) → Epic (self-contained) → Story (precise)
